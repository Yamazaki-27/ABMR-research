なぜ GitHubを使うのか<br>
<br>
開発を進める中で、こんな経験はありませんか？<br>
<br>
「以前うまくいっていたのに、どこを変えてから動かなくなったのかわからない」<br>
「自分が変更したファイルを、別のメンバーが古いバージョンで上書きしてしまった」<br>
「実験のメモをExcelで管理していたが、どれが最新版かわからなくなった」<br>
「担当者が変わったら、これまでの経緯が誰もわからなくなった」<br>
<br>
これらはすべて、変更履歴が管理されていないことで起きる問題です。<br>
GitHubはこれを解決するための仕組みです。<br>
<br>
✅ GitHubでできること<br>
課題GitHubによる解決いつ・誰が・何を変えたか不明全ての変更が日時・氏名つきで自動記録されるファイルの上書き事故変更の競合を検知して防ぐ仕組みがある「前の動いていたバージョン」に戻せない任意の時点に1クリックで戻せる実験的な変更が本番に影響してしまうブランチ機能で安全に試せる作業スペースを作れる担当者が変わると知識が消えるコードと一緒にコメント・経緯が残り続ける離れた場所のメンバーと共有できないインターネット経由でリアルタイムに共有できる<br>
<br>
💡 開発においてGitHubは「保険」であり「記憶装置」である<br>

プログラムの開発は、変更の積み重ねで成り立っています。<br>
AMRは、小さな改善を何百回と繰り返して完成に近づいていきます。<br>
その一つひとつの「試行錯誤の記録」が、チームの財産になります。<br>
<br>
GitHubを使うことは、開発の効率を上げるだけでなく、<br>
チームの知識と経験を会社の資産として積み上げることでもあります。<br>

## 🚀 GitHubをどう使うか

このリポジトリでは以下のルールで運用します。

| やること | 意味 |
|---|---|
| **commit（コミット）** | 変更内容を記録する（セーブポイントを作る） |
| **push（プッシュ）** | 自分の変更をGitHubに反映する |
| **pull（プル）** | 他メンバーの変更を自分の環境に取り込む |
| **branch（ブランチ）** | 本番に影響せず試せる作業スペースを作る |

> **困ったときは一人で抱え込まず、チームに声をかけてください。**
> このリポジトリはチームの共有財産です。
