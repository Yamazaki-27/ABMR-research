# 🤖 なぜ GitHubを使うのか<br>
<br>
これまでの開発では、誰もがこんなことを感じてきたと思います<br>
<br>
「書庫の開発資料ファイルに書いてあるはずだけど、どこにあるのか？探せない」<br>
「大事なプログラミングのノウハウが社内に残っていない」<br>
「シーケンサーは、他人が理解するのは難しい」<br>
「実験のメモをExcelで管理していたが、どれが最新版かわからなくなった」<br>
「担当者が変わったら、これまでの経緯が誰もわからなくなった」<br>
<br>
これらはすべて、変更履歴が管理されていないことで起きる問題です。<br>
GitHubはこれを解決するための仕組みです。<br>
<br>
✅ GitHubでできること<br>
課題GitHubによる解決いつ・誰が・何を変えたか不明全ての変更が日時・氏名つきで自動記録されるファイルの上書き事故変更の競合を検知して防ぐ仕組みがあります。<br>
<br>
「前の動いていたバージョン」に戻せない<br>
　　└──任意の時点に1クリックで戻せる<br>
実験的な変更が本番に影響してしまう<br>
　　└──ブランチ機能で安全に試せる<br>
　　└──作業スペースを作れる<br>
担当者が変わると知識が消える<br>
　　└──コードと一緒にコメント<br>
　　└──経緯が残り続ける<br>
　　└──離れた場所のメンバーと共有できないインターネット経由でリアルタイムに共有できる<br>
<br>
💡 開発においてGitHubは「保険」であり「記憶装置」です<br>
<br>
プログラムの開発は、変更の積み重ねで成り立っています。<br>
AMRは、小さな改善を何百回と繰り返して完成に近づいていきます。<br>
その一つひとつの「試行錯誤の記録」が、チームの財産になります。<br>
<br>
GitHubを使うことは、開発の効率を上げるだけでなく、<br>
チームの知識と経験を会社の資産として積み上げることでもあります。<br>

## 🚀 GitHubをどう使うか

このリポジトリでは以下のルールで運用します。

| やること | 意味 |
|---|---|
| **commit（コミット）** | 変更内容を記録する（セーブポイントを作る） |
| **push（プッシュ）** | 自分の変更をGitHubに反映する |
| **pull（プル）** | 他メンバーの変更を自分の環境に取り込む |
| **branch（ブランチ）** | 本番に影響せず試せる作業スペースを作る |

📌 重要📌
議事録、仕様書、メモといった文書は、同時編集による競合の可能性が極めて低いです。<br>
なので、各自のタイミングで、mainでpushしてかまいません。<br>

**困ったときは一人で抱え込まず、チームに声をかけてください。**
このリポジトリはチームの共有財産です。<br>
